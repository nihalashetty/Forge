# Forge Uncommitted-Changes Audit — Generative UI Components & End-User Identity

This audit covers the uncommitted changes introducing (a) the end-user identity prompt block, (b) UI components as agent widget-tools, and (c) the chat-rendering of interleaved text + components. Findings below are de-duplicated from the verified set; where two findings describe the same root cause, they are merged.

## Part 1 — User-Reported Priorities (root cause + fix)

### Priority A — Knowledge-base / FAQ Q&A became unreliable after components landed

**Root cause (confirmed):** A newly added `[END USER]` block is appended to the system prompt on any identity-carrying run. It instructs the model to "never take actions on, or reveal data about, anything they are not entitled to" — but supplies **no positive entitlement list** (the `end_user` shape is fully generic, all fields optional). With an unscoped prohibition and no scope, a cautious model treats general KB/FAQ content as possibly-restricted and refuses or hedges. Before this change there was no identity block, so KB answers were never gated this way. It is intermittent because it only fires when `end_user` is present and depends on the model judging the answer "sensitive." Verified at `apps/api/forge/nodes/agent_node.py:133-141`; `end_user` is generic in `apps/api/forge/engine/context.py:52-55` and `apps/api/forge/schemas/dto.py:137-141`; propagated from runs (`services/runs.py:153,265,318`) and the Playground "Acting as" field (`apps/web/components/screens/playground.tsx:198-201`, whose own placeholder `{"id":"u_1","roles":["admin"]}` carries no entitlements). The platform's dominant use case is KB/FAQ grounding (`tools/builtin.py` exposes `search_knowledge_base`/`lookup_faq`), so the prohibition lands on exactly the flow that should answer freely.

**Severity:** High user impact (verified medium as a code defect — behavioral/probabilistic, model-dependent, no deterministic refusal path).

**Fix** — `apps/api/forge/nodes/agent_node.py`, in `_common_kwargs`, rewrite the `eu_block` so the restriction is positively scoped:
- Affirm that public/KB/FAQ content is always answerable.
- Limit withholding to data specific to *other* users/accounts. Example replacement third sentence: *"General product, FAQ, and knowledge-base information is available to everyone — always answer it. Only withhold data that is specific to other users or accounts this user is not entitled to act on or see."*
- Preferably, only append the restriction sentence when `end_user` actually contains a non-empty `roles`/`entitlements` list; when those are absent, emit identity-awareness only (the "you are assisting {eu}" sentence) with no prohibition — this removes the chilling instruction from the common no-entitlements case entirely.

### Priority B — Component sometimes renders as a widget, sometimes as plain prose

This symptom has **two distinct causes** — one inherent (model variance) and one deterministic (a prompt conflict). Fix the deterministic one; mitigate the variance.

**B1 — Deterministic: OUTPUT_STYLE vs COMPONENT_STYLE contradict each other.** When components are attached, `_build_prompt` appends **both** `OUTPUT_STYLE` (which says "use GFM tables for comparisons or structured data," `agent_node.py:23-29`) and `COMPONENT_STYLE` (which says "Do NOT also write the same data as text," `agent_node.py:35-41`). For the exact structured-data case a table/card component targets, the model is told opposite things — a self-inflicted, fixable cause of flip-flopping. Confirmed: both appended together at `agent_node.py:51-53`.

**B2 — Inherent: the component is just an optional tool.** A component is materialized as an ordinary `StructuredTool` (`apps/api/forge/tools/components.py:58`) and appended to the same flat `tools` list as everything else (`agent_node.py:121`). There is no `tool_choice`/forced-tool mechanism anywhere in the package, so whether the widget fires is a free ReAct-loop decision steered only by the soft `COMPONENT_STYLE` suffix. Full determinism is not achievable with a sampling LLM.

**Severity:** Medium (quality/UX; workaround exists).

**Fix** — `apps/api/forge/nodes/agent_node.py` (`_build_prompt`, lines 44-54):
1. **(Load-bearing)** When `config.get("components")` is truthy, do **not** emit the unconditional OUTPUT_STYLE "GFM tables for structured data" clause — either swap in a components-aware OUTPUT_STYLE variant that drops that clause, or order COMPONENT_STYLE to explicitly override it.
2. Make COMPONENT_STYLE imperative for the canonical case: *"If a table/card/form component exists that fits the data, you MUST call that component tool with the data as its props instead of writing the data as prose; do not also restate it as text."*
3. (Optional) Expose a per-agent low-temperature/determinism knob via `model_params` (`resolve_model` already reads it, `agent_node.py:125`) for component-heavy agents.

### Priority C — Components should flow WITH the text, not appear as a separate card

**Root cause (confirmed):** Assistant turns render each text part as a **bordered chat bubble** and each component as a **standalone iframe card**, stacked with gaps — so prose reads as a bubble and the component reads as a detached card below it. In `MessageBlock` (`playground.tsx`) and `TestMessage` (`WorkflowTestPanel.tsx`): `bubble()` wraps assistant text in a box with `background var(--bg-2)`, `1px solid var(--line)`, `borderRadius 12`, `padding 10px 13px` (`playground.tsx:316-321`; test panel uses `var(--bg-3)` at `:215`); `renderComp()` emits a bare `<ComponentRenderer>` iframe (`playground.tsx:322-328`); the container is `.col gap2` (8px gap). The part *ordering* is correct — this is purely chrome/layout, not data. `.md` already supplies its own block margins (`globals.css:233-251`), so assistant text needs no padded container.

**Severity:** Medium (cosmetic; content is usable and correctly ordered).

**Fix** — in both `MessageBlock` (`apps/web/components/screens/playground.tsx`) and `TestMessage` (`apps/web/components/canvas/WorkflowTestPanel.tsx`):
1. Keep `user = role === "user"` and the existing colored `bubble()` for **user** turns only.
2. For **assistant** turns, render text parts as bare `<Markdown>` with no background/border/borderRadius/padding — just inherit `fontSize:14/lineHeight:21px` (13/19px in the test panel) on the content wrapper to match old bubble typography. Keep the streaming cursor appended to the last text block.
3. Render each component inline via `<ComponentRenderer>` immediately after the preceding text, with a small consistent top margin (~8px), so text → component → text reads as one reply under the single avatar.

**Related (do alongside C):** Even after removing bubble chrome, the component still *pops in only at finalize* (see I3 below) — this will still feel like a late, separate card unless the in-flight turn renders from live parts.

## Part 2 — Other Confirmed Issues (grouped by severity)

### HIGH

**H1 — Components are silently dropped on every non-Playground channel.** The widget renders only via a custom-stream `component` frame (`components.py:37-50`). The SSE path captures it with `astream(stream_mode=[…,"custom"], subgraphs=True)` (`services/runs.py:181-186`), but the dispatcher path (`services/dispatch.py:33,50` → `run_to_completion`) uses `await graph.ainvoke(...)` (`runs.py:271`), which never surfaces custom-stream items, so the frame is discarded. `run_to_completion` returns only `_last_ai_text(...)` (`runs.py:285`) — the model's ack minus any restated content (COMPONENT_STYLE told it not to restate). Channel adapters only send when `reply_text` is truthy (`channels.py:153,175`), so an empty answer = **no reply sent**. Affects webhook, schedule, email, Teams, app-event polling, and evals. Deterministic.
**Fix:** In `run_to_completion`, replace `ainvoke` with an `astream(stream_mode=["custom"], subgraphs=True, …)` loop that collects `channel=="component"` payloads, return them as `"components": [...]` (`runs.py:282-287`); add graceful degradation for text-only surfaces (channel-capability flag in the prompt, or serialized fallback) and never send an empty reply when a component was the only output. Reference pattern: the SSE `astream` at `runs.py:181`.

**H2 — Final answer text (`done.answer`) and error frames are dropped when a component is present.** On finalize the code branches on `hasComp`: if any component arrived it builds the message from `parts` only and **never references `finalAnswer`** (`playground.tsx:110-116`; `WorkflowTestPanel.tsx:140-146`). But `finalAnswer` is the authoritative `done.answer` (`runs.py:220-229`, covering text from non-streaming nodes — and components *can* be emitted from the non-streaming `emit_event` node, `nodes/data.py:140-153`). Worse: an `error` after a component frame sets `finalAnswer="⚠ …"` (`playground.tsx:100`) but `hasComp` is true, so the user sees a stale component and **no failure notice**.
**Fix:** In both finalize blocks, do not let `hasComp` suppress `finalAnswer`. Always surface error/interrupt text regardless of `hasComp` (push it as a trailing text part). When `hasComp` and `finalAnswer` is non-empty and differs from the concatenated streamed text, append a text part built from `finalAnswer`.

**H3 — Components rendered mid-stream are invisible until finalize; a component-only reply shows an empty "…" bubble.** `parts` is a plain local array, not React state (`playground.tsx:63`; `WorkflowTestPanel.tsx:78`); component frames push only into it (`:91-92` / `:114-115`) with no state update. The live bubble renders `content={streaming}` with no `parts` prop (`:236` / `:178`), so the widget appears only at finalize. If the agent calls the component tool with no surrounding text (the designed common case — the ack says "Do not restate its contents"), `streaming` stays `""` while `running` is true and the live block renders the literal "…" placeholder with the component hidden.
**Fix:** Lift in-flight `parts` into state (`liveParts`), mirror every text/component mutation in the SSE handler into `setLiveParts([...parts])`, and pass `parts={liveParts}` to the streaming `MessageBlock`/`TestMessage`. Keep the cursor on the trailing text part only; reset `liveParts` at the start of `send()` and in `finally`. (This also resolves the Priority C "late card" reflow.)

> Note: findings "frame arrives before compDefs loaded" and "missing component def renders nothing" are merged into **M5** below (same null-render root cause); they were independently rated medium on verification.

### MEDIUM

**M1 — Unguarded `select(Component)` aborts the entire run.** Only the per-row `build_component_tool` call is wrapped in try/except (`runtime.py:124-134`, "skip a broken component; don't break the run"); the `session.execute(select(Component)…)` that loads the rows (`runtime.py:114-122`) is **bare**, unlike the MCP block below which guards per-server (`runtime.py:152-156`). Any DB hiccup, RLS denial, or JSON-column deserialization error propagates out of `build_compile_context` — which in `stream()` is awaited *before* the error handlers and has no outer try/except, leaving the run stuck in `status="running"` with no error frame. Defeats the author's stated intent (guard is on the wrong line).
**Fix:** Wrap the `select(Component)` execution in try/except that logs and falls back to `comp_rows = []`, matching the MCP pattern. Optionally also guard the `build_compile_context` call in `stream()` so any context-build failure emits an error frame and sets `run.status="error"`.

**M2 — Duplicate component names (or component-vs-tool name) collide silently → wrong/shadowed widget.** Component tools use the component `name` verbatim as the LLM tool name (`components.py:27,59`) and are concatenated into the flat `tools` list (`agent_node.py:109-121`). `name` has no DB unique constraint (`entities.py`) and no validation in `ComponentService.create/update` or the router. LangChain's `ToolNode` does `self._tools_by_name[tool_.name] = tool_` (last-wins overwrite, no error), and the model is bound to the deduplicated set — so the earlier same-named tool/component is **permanently shadowed**; the survivor emits a frame whose `component_id` may point at the wrong template. The workflow validator never inspects the resolved tool-name set. Deterministic last-wins (not random).
**Fix:** (1) Add a DB `UniqueConstraint(tenant_id, project_id, name)` to `Component` (`entities.py`) + migration, plus a pre-check in `ComponentService.create/update` (`services/components.py`) raising 409. (2) In `_common_kwargs` (`agent_node.py`), after assembling `tools`, detect duplicate `.name` across regular + knowledge + MCP + component tools and rename/raise; surface a warning via `services/validation.py`. (Namespacing component tool names with a `ui_`/`component_<id>` prefix in `components.py:58-60` is the lowest-friction one-line guard against the component-vs-tool case.)

**M3 — Component `name` is not validated against the LLM tool-name charset.** `ComponentCreate.name`/`ComponentUpdate.name` are bare `str` (`routers/components.py:23,36`), stored verbatim (`services/components.py:41,58`), passed straight as the StructuredTool name (`components.py:59`). A name like "My Table" is accepted but rejected by real providers (`^[a-zA-Z0-9_-]+$`), failing at request time (the fake test model masks it). This is a **regression of an existing guard**, not a mirror: the Tool router *does* validate via `validate_against_id(cfg, "forge/tool")` against `forge/common#/$defs/Identifier` (`packages/schemas/forge/common.json:41-45`); the component surface dropped it and has no `forge/component.json`.
**Fix:** Add a Pydantic pattern `name: str = Field(pattern=r"^[a-zA-Z0-9_-]{1,64}$")` to `ComponentCreate`/`ComponentUpdate` (`routers/components.py`); cleanest is a `forge/component.json` contract that `$ref`s the shared `Identifier` and is enforced via `validate_against_id` in create/update.

**M4 — Streamed text/components produced before an approval interrupt are discarded (Playground only).** On `interrupt`, the loop does `if (interrupted) { setStreaming(""); setRunning(false); return; }` (`playground.tsx:103`) **before** the finalize block (`:110-116`), so any component already pushed to `parts` is thrown away; `resume()` (`:139-156`) rebuilds the assistant message only from `res.messages` text with no component handling, so the widget vanishes permanently. The parallel `WorkflowTestPanel.tsx` does *not* early-return and preserves parts — confirming the Playground branch is the defective one.
**Fix:** Before the early `return`, flush accumulated `parts`/`buffer` into `msgs` using the same finalize logic (commit a parts-based message when `hasComp || buffer.trim()`). Optionally harden `resume()`/backend resume to carry forward prior parts.

**M5 — Missing/uncached component def renders `null` → blank assistant turn (no fallback).** `renderComp` does `const def = compDefs[inst.component_id]; if (!def) return null;` (`playground.tsx:323-324`; `WorkflowTestPanel.tsx:221-222`). `compDefs` is fetched once per project/workflow, best-effort with `.catch(() => {})`, and the SSE frame carries only ids+props, **not** html/css (`components.py:40-50`). On a cache miss — component created after page load (cache never refreshed), a swallowed `listComponents` failure, or a fast pre-fetch race — the lone component part renders nothing; `hasComp` stays true so the text-fallback branch is skipped → an assistant turn with just an avatar and an empty column. (This merges the two separately-reported findings on the same null-render path.)
**Fix:** Replace `return null` with (a) a lazy `api.getComponent(project.id, inst.component_id)` fetch (it exists, `api.ts:265`) showing a "loading component…" placeholder and merging into `compDefs`, and (b) on hard failure, a visible muted fallback (`<pre>` of props or `Component "<name>" unavailable`). Stop silently swallowing the `listComponents` failure (at least log it). Apply identically in both files.

### LOW

**L1 — Props undergo only shallow Pydantic coercion, not real `props_schema` validation.** `args_schema` is built by `build_args_schema_from_jsonschema` (`components.py:31` → `rest.py:98-110`), which maps only top-level scalars; nested object/array item shapes are unvalidated, unknown types fall back to `str`, non-required fields default to `None`, and `dict(kwargs)` (incl. `None`s) is forwarded verbatim to Mustache (`component-renderer.tsx:44`). The common path degrades gracefully (Mustache treats null/empty sections as falsy), so the blank-widget/"intermittent prose" framing is overstated — real impact is a structurally-wrong *nested* value rendering wrong. Genuine but low.
**Fix:** In `components.py` `_call`, before emitting the frame: drop `None`-valued keys, then validate cleaned props against `cfg["props_schema"]` with a real JSON-Schema validator (e.g. `jsonschema.Draft202012Validator`); on failure return an actionable tool-error naming the bad field so the model re-calls the widget. Optionally extend the args-schema builder to recurse into object/array item schemas. Leave the `stream_writer=None` ack-only path as-is.

**L2 — Component get/update/delete are tenant-scoped only (cross-project access within a tenant).** `ComponentService.get` filters by `tenant_id + id`, ignoring `project_id` (`services/components.py:27-31`); the routes take `project_id` in the path but never pass it (`routers/components.py:73,82,90`). A caller can read/update/delete project A's component via project B's URL (same tenant). Not a regression — every service `get()` in the codebase shares this pattern; tenant isolation (the primary boundary) is intact.
**Fix:** Add `project_id` to `ComponentService.get`'s signature and `Component.project_id == project_id` to the where clause; pass `project_id` from the three routes. Optionally add `Depends(require_role("editor"))` on POST/PATCH/DELETE to align with channels/evals.

**L3 — `ComponentUpdate` cannot change `kind`; `update()` bumps version on every PATCH (incl. no-ops).** `ComponentUpdate` omits `kind` and `update()` has no `kind` param (`routers/components.py:34-43`; `services/components.py:51-56`), so kind is immutable post-create (no live impact — only `"html"` exists today; `"declarative"` is future). `comp.version += 1` is unconditional (`services/components.py:77`), and the router uses `exclude_unset=True`, so an empty PATCH still bumps the version that keys client render-cache invalidation (id@version) → needless cache busts.
**Fix:** Bump `version` only when a template-affecting field (`html`/`css`/`props_schema`/`actions`) actually changed value. Defer adding `kind` to update until a second valid kind exists.

**L4 — Untrusted `end_user` is embedded verbatim into the system prompt as "authoritative"; whole-blob truncation can malform JSON.** `_json.dumps(end_user, default=str)[:1500]` (`agent_node.py:135`) is a character-level slice that can cut JSON mid-token, and any `end_user` string field becomes part of the trusted prompt (`dto.py:141` is free-form `dict[str, Any]`). Low because the documented threat model has the host set this server-side from a verified session.
**Fix:** Surface only a whitelisted, per-field length-clamped subset (`id`, `display_name`, `roles`, `entitlements`), then re-serialize that subset (so the embedded JSON stays well-formed). Reinforce/validate the server-set contract with a small Pydantic model rather than an arbitrary dict.

## Part 3 — Ordered Fix Plan

Sequencing favors load-bearing, cross-cutting fixes first; backend before frontend where the frontend depends on backend shape; cheap high-leverage prompt edits early.

1. **Fix the KB/FAQ regression (Priority A)** — rewrite the `eu_block` in `agent_node.py` to positively scope entitlements (and gate the prohibition on a non-empty roles/entitlements list). *Directly addresses the most user-visible breakage; one-file change.*

2. **Resolve the prompt conflict (Priority B1)** — in `_build_prompt`, suppress the OUTPUT_STYLE "GFM tables for structured data" clause when components are attached, and make COMPONENT_STYLE imperative (B2 mitigation). *Same file as #1; do together.*

3. **Stop dropping components on non-Playground channels (H1)** — convert `run_to_completion` from `ainvoke` to a custom-stream `astream` loop and return `components`; add text-only-surface degradation. *Backend correctness; unblocks every channel and establishes the shape channel adapters consume.*

4. **Guard the component-load query (M1)** — wrap `select(Component)` in try/except → `[]`. *Small, prevents total run failure; do alongside #3 in `runtime.py`/`runs.py`.*

5. **Enforce component name validity + uniqueness (M3, M2)** — add the charset pattern (or `forge/component.json` contract) and the `(tenant, project, name)` unique constraint + compile-time collision guard. *Prevents silent wrong-widget and provider request failures before they reach users.*

6. **Live-render the in-flight turn from stateful parts (H3) + flow layout (Priority C)** — lift `parts` into `liveParts`, render the streaming block from it, and remove assistant bubble chrome so text and components flow as one reply. *These two are the same render path; doing them together avoids touching `MessageBlock`/`TestMessage` twice and eliminates both the empty-"…" bubble and the end-of-turn reflow.*

7. **Preserve authoritative answer/error when a component is present (H2)** — in both finalize blocks, append `finalAnswer`/error as a text part instead of letting `hasComp` suppress it.

8. **Resilient missing-def rendering (M5)** — replace `return null` with lazy `getComponent` + visible fallback; stop swallowing the `listComponents` error. *Same two files as #6/#7.*

9. **Flush parts on interrupt in Playground (M4)** — mirror `WorkflowTestPanel`'s non-early-return behavior.

10. **Validate props against `props_schema` + strip `None`s (L1).**

11. **Remaining low-severity hardening (L2 cross-project scoping, L3 version-bump/kind, L4 end_user whitelisting)** — batch as cleanup.

# Confirmed findings

### 1. [medium] (knowledge-regression) [END USER] entitlement block causes over-cautious refusals of KB/FAQ answers
- files: apps/api/forge/nodes/agent_node.py, apps/api/forge/engine/context.py
- fix: In apps/api/forge/nodes/agent_node.py (_common_kwargs, the eu_block string), make the restriction positively scoped so general knowledge is never gated. Replace the current third sentence with wording that (a) affirms public/KB/FAQ content is always answerable and (b) limits withholding to data specific to OTHER users/accounts. Example: "[END USER] You are assisting this authenticated end user, provided by the host application — treat it as authoritative: {eu}. General product, FAQ, and knowledge-base information is available to everyone — always answer it. ### 2. [medium] (component-render-consistency) Primary cause is model-choice variance, not a bug: the component is exposed as an ordinary, optional tool the agent may or may not call
- files: apps/api/forge/nodes/agent_node.py, apps/api/forge/tools/components.py
- fix: Resolve the deterministic part first (the prompt conflict), which is the load-bearing fix, then strengthen steering. In _build_prompt (apps/api/forge/nodes/agent_node.py:44-54): (1) When config.get("components") is truthy, do NOT emit the unconditional OUTPUT_STYLE "GFM tables for comparisons or structured data" clause — either swap in a components-aware OUTPUT_STYLE variant that drops the structured-data-as-table guidance, or reorder so COMPONENT_STYLE takes precedence and explicitly overrides it. The current behavior appends both and tells the model opposite things for the same input. (2) Make COMPONENT_STYLE imperative for the canonical case, e.g. "If a table/card/form component exists that fits the data, you MUST call that component tool with the data as its props instead of writing the data as prose; do not also restate it as text." (3) Optionally, for component-heavy agents, expose a per-agent low-temperature / determinism knob via model_params (resolve_model already takes config.get("model_params"), agent_node.py:125) to reduce run-to-run flip-flopping. Full determinism is not achievable with a sampling model and is out of scope; (1) is the concrete, code-grounded change that most reduces the inconsistency the user reported.

### 3. [high] (component-render-consistency) Components are silently dropped on every non-Playground channel: run_to_completion uses ainvoke (no custom stream)
- files: apps/api/forge/services/runs.py, apps/api/forge/services/dispatch.py, apps/api/forge/tools/components.py
- fix: In RunService.run_to_completion, consume the graph with the custom stream instead of ainvoke so component frames are captured and returned to channel adapters. Replace `await graph.ainvoke(run.input, config, durability=settings.run_durability)` (runs.py:271) with an astream loop, e.g.: `components = []; async for ns, mode, chunk in graph.astream(run.input, config, stream_mode=["custom"], subgraphs=True, durability=settings.run_durability): if mode == "custom" and isinstance(chunk, dict) and chunk.get("channel") == "component": components.append(chunk.get("payload"))`. Then add `"components": components` to the returned dict (runs.py:282-287). Channel adapters can then render/attach them where the surface supports rich UI. For text-only surfaces (email/Teams) that cannot render widgets, add graceful degradation: either pass a channel capability flag into the agent prompt so it produces a textual version when no rich-UI surface exists, OR have the adapter fall back to a serialized/summary representation of the component props so the reply is never empty. At minimum, ensure the channel reply is never empty when a component was the only output (e.g. if `not reply_text and components`, synthesize a textual fallback), and document that live component rendering currently requires a rich-UI (SSE/Playground) surface. Note the SSE astream at runs.py:181 is the correct reference pattern (it already uses stream_mode including "custom" with subgraphs=True).

### 4. [medium] (component-render-consistency) Component frame can arrive before compDefs is loaded → renderComp returns null and the widget silently vanishes
- files: apps/web/components/screens/playground.tsx, apps/web/components/canvas/WorkflowTestPanel.tsx
- fix: Make the renderer resilient to cache misses instead of returning null. When compDefs[inst.component_id] is missing: (1) lazily fetch the def via api.getComponent(project.id, inst.component_id), showing a lightweight 'loading component…' placeholder while it resolves, and merge the result into compDefs so it renders once available; (2) on a hard 404/fetch failure, render a visible fallback (e.g. a <pre> of inst.props or the component name) rather than null, so the assistant turn is never blank. Also stop swallowing the listComponents failure silently (`.catch(() => {})`) — at minimum log it, since a failed initial load leaves every component a permanent cache miss. Apply identically in both playground.tsx renderComp (323-324) and WorkflowTestPanel.tsx renderComp (221-222). Optionally, to make the cache robust against stale/cross-project state, key it by project id or refresh it before building the assistant message.

### 5. [low] (component-render-consistency) Props are coerced/dropped by the auto-generated Pydantic args schema; a failed/empty props object can make the widget render blank or the model fall back to prose
- files: apps/api/forge/tools/components.py, apps/api/forge/tools/rest.py, apps/web/components/component-renderer.tsx
- fix: In components.py _call, before emitting the frame, (a) drop None-valued keys from kwargs so optional props fall back to the template's own Mustache defaults / falsy-section behavior instead of being forwarded as null, and (b) validate the cleaned props against cfg['props_schema'] with a real JSON-Schema validator (e.g. jsonschema.Draft202012Validator) and, on failure, return an actionable tool-error string naming the offending field/expected shape so the model re-calls the widget with corrected args rather than potentially answering in prose. Optionally extend build_args_schema_from_jsonschema to recurse into object/array item schemas so structured args (like rows: [{label, value}]) round-trip faithfully, and consider strict typing for integer/number/boolean fields to avoid silent lax coercion.
